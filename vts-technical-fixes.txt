SECTION 1 — HIGH PRIORITY (fix first, these affect crawling and indexing)
===================================================================

1.1 Internal 404 Errors — 2 broken internal URLs
----------------------------------------------------------------------
Issue: 2 internal pages return a 404 error. One is confirmed to be
/career-transition-program/ which is linked from the homepage. The
second is a Cloudflare email protection URL that has 99 internal links
pointing to it across the site — this is the most impactful single fix.

Fix:
- Find the Cloudflare email obfuscation script that is generating the
  broken URL. This is almost certainly coming from Cloudflare's email
  address obfuscation feature being applied to a mailto: link somewhere
  in the theme or a plugin. Disable Cloudflare email obfuscation in the
  Cloudflare dashboard (Speed > Optimization > Email Address Obfuscation,
  turn it off), or manually replace the affected mailto links with
  properly encoded versions.
- For /career-transition-program/: check if the page exists as a draft
  or was deleted. Either restore it, create a redirect to the correct
  page (/career-transition-program/ → the appropriate live program page),
  or update all internal links pointing to it.
- After fixing, confirm both URLs return 200 in a re-crawl.

1.2 HTTP URL in Crawl — 1 internal HTTP URL
----------------------------------------------------------------------
Issue: 1 internal URL is being served or linked as HTTP instead of HTTPS.
This triggers a "Not Secure" warning in browsers.

Fix:
- In WordPress Settings > General, confirm both WordPress Address and
  Site Address are set to https://veteranstransitionsupport.org (not http).
- Check for any hardcoded http:// links in the theme, page builder content,
  or widgets. Use a plugin like Better Search Replace to do a database-wide
  search for http://veteranstransitionsupport.org and replace with
  https://veteranstransitionsupport.org.
- Confirm the hosting server has an HTTPS redirect rule (301 redirect all
  HTTP to HTTPS) in .htaccess or the server config.
- WP Rocket is already installed — confirm "Disable Google Fonts" and
  "Load fonts locally" are not introducing any HTTP resource loads.

1.3 Pagination Issues — 37-38 affected URLs (High Priority)
----------------------------------------------------------------------
Issue: 37 pagination URLs are non-indexable and 38 pagination URLs are not
properly linked in anchor tags. These are generated by The Events Calendar
Pro's archive views (month view, list view, day view, category archives).
This wastes crawl budget and means Google may not properly crawl and index
paginated event content.

Root cause investigation required:
- Go to The Events Calendar > Settings > General in WordPress and check
  how archive URLs and pagination are configured.
- Check whether a "noindex" tag or a canonical pointing back to the first
  page is being applied to paginated event archive pages (e.g.
  /events/page/2/, /events/month/2026-04/, etc.).
- Check if The Events Calendar is generating pagination URLs that do not
  have proper <a href> links (rel="next" / rel="prev" links in the head
  that are not matched by actual clickable anchor tags on the page).

Fix:
- If event archive pages beyond page 1 are canonicalising to page 1,
  change them to self-referencing canonicals.
- Ensure paginated event archive URLs are linked with proper <a> tags in
  the page content so Google can follow them.
- If the events plugin is generating many low-value date-based archive
  pages (e.g. /events/2024/01/, /events/2023/12/), consider setting those
  to noindex while keeping the main /events/ page and monthly views
  indexable.
- After fixing, confirm pagination URLs are indexable and properly linked
  in a re-crawl.

1.4 Canonicalised Pages — 37 affected URLs
----------------------------------------------------------------------
Issue: 37 pages have canonical tags pointing to a different URL. This is
related to the pagination issue above. Verify that these canonicals are
intentional and pointing to the correct destination. The most common
mistake is pagination pages canonicalising to page 1 — this tells Google
to ignore the paginated pages entirely.

Fix:
- Export the full canonical report from Screaming Frog.
- For each canonicalised URL, verify the canonical destination is correct
  and intentional.
- Any pagination canonical pointing back to the root /events/ page should
  be updated to a self-referencing canonical.
- Any duplicate content pages (where a second version of a page exists due
  to a previous rebuild) should have the weaker version redirected via 301
  to the canonical version, not just canonicalised — 301 redirects are
  cleaner than canonicals for true duplicates.

SECTION 2 — ON-PAGE SEO (do after Section 1 root cause work is done)
===================================================================

2.1 Duplicate Page Titles — 21 pages affected
----------------------------------------------------------------------
Issue: 21 pages share the same page title. The main offender is almost
certainly the generic "Military To Civilian Career Transition & Job
Training | San Diego" title being applied to multiple pages. When multiple
pages share a title, Google cannot distinguish them and ranking signals
are split.

Fix:
- Export the duplicate titles report from Screaming Frog.
- For each affected page, write a unique, descriptive page title that
  includes the specific topic of that page and a relevant keyword.
  Keep titles under 60 characters (560 pixels) where possible.
- Examples of what unique titles should look like:
  Homepage: "Military to Civilian Career Training | VTS San Diego"
  OSHA 30: "OSHA 30 Training for Veterans | VTS San Diego"
  LSSGB: "Lean Six Sigma Green Belt for Veterans | VTS"
  Events: "Upcoming Free Veteran Training Classes | VTS"
- Use Rank Math or Yoast to update titles — do not change the page slug
  or URL when editing titles.

2.2 Page Titles Too Long — 17 pages over 60 characters / 561 pixels
----------------------------------------------------------------------
Issue: 17 page titles exceed Google's recommended display length and will
be truncated in search results. The most important words get cut off.

Fix:
- Review each title flagged in Screaming Frog.
- Shorten to under 60 characters while keeping the primary keyword and
  brand name. Move the brand name to the end: "Program Name | VTS" rather
  than "VTS | Veterans Transition Support | Program Name".

2.3 Missing H1 Tags — 4 pages with no H1
----------------------------------------------------------------------
Issue: 4 pages have no H1 tag. These pages are invisible to Google in
terms of topic relevance. Confirmed pages likely include /events/,
/testimonies/, and possibly /osha-30-safety-training/ or /donate/.

Fix:
- In Elementor, open each affected page and add a proper H1 heading at
  the top of the page content that clearly describes what the page is about.
- The H1 should be the most prominent heading visible to users when the
  page loads — it does not need to be styled large if the design calls
  for a different visual treatment, but it must exist in the HTML as an
  actual H1 element (not styled text or a div).
- Confirm H1 presence with a re-crawl.

2.4 Duplicate Meta Descriptions — 10 pages affected
----------------------------------------------------------------------
Issue: 10 pages share identical meta descriptions. Google ignores duplicate
meta descriptions and writes its own snippet, which is usually worse than
a crafted one.

Fix:
- Export the duplicate meta descriptions report.
- Write unique meta descriptions for each affected page.
- Target 140-155 characters (around 920 pixels).
- Each description should describe what that specific page offers and
  include a call to action where appropriate.
  Example for OSHA 30 page: "Live OSHA 30 training for veterans on Zoom
  and in person. Taught by OSHA Authorized Outreach Trainers in San Diego.
  Enroll today."

2.5 Missing Meta Description — 1 page with no meta description
----------------------------------------------------------------------
Fix: Add a unique meta description to this page via Rank Math or the page
editor. Identify the page from the Screaming Frog missing meta description
export.

2.6 Meta Descriptions Too Short — 8 pages below 70 characters
----------------------------------------------------------------------
Fix: Expand these descriptions to 140-155 characters to take full
advantage of the available display space in Google search results.

2.7 Meta Description Too Long — 1 page over 985 pixels
----------------------------------------------------------------------
Fix: Shorten this meta description to under 155 characters so it does not
get truncated in search results.

2.8 H2 Heading Issues — Multiple pages
----------------------------------------------------------------------
Three separate H2 issues found:
- 16 pages have duplicate H2s (the same H2 text appears on multiple pages)
- 38 pages have multiple H2s (not an error, but worth reviewing for structure)
- 7 pages have non-sequential H2s (an H2 appears before an H1, or an H3
  is used where an H2 should be)
- 4 pages are missing H2s entirely

Fix:
- For duplicate H2s: update the heading text on each page so each page
  has a unique and descriptive H2 that reflects the page content.
- For non-sequential headings: open each flagged page in Elementor and
  correct the heading levels so they flow H1 → H2 → H3 in logical order.
  Never skip heading levels.
- For missing H2s: add at least one descriptive H2 to each content page
  to help users and search engines understand the page structure.

2.9 H1 Over 70 Characters — 8 pages
----------------------------------------------------------------------
Fix: Shorten H1 text on these pages to be more concise. The H1 should be
a clear, direct statement of what the page is about. Trim filler words.

2.10 One Page Has Multiple H1s
----------------------------------------------------------------------
Fix: Find the page with multiple H1 tags (use Screaming Frog H1 Multiple
export), open it in Elementor, and convert all but the primary H1 to H2

SECTION 3 — IMAGE OPTIMIZATION
===================================================================

3.1 Images Missing Alt Text — 51 images
----------------------------------------------------------------------
Issue: 51 images have an empty alt attribute. These are invisible to
screen readers and search engines.

Fix:
- Export the missing alt text report from Screaming Frog to get the list
  of image URLs and which pages they appear on.
- For each image, add descriptive alt text that describes what the image
  shows in the context of the page.
- For decorative images that convey no meaning: set alt="" (empty string)
  rather than leaving the attribute missing.
- For images of people, instructors, program sessions: describe what is
  shown, e.g. "Russell Levy teaching OSHA 30 class at VTS in San Marcos CA"
- Do NOT stuff keywords into alt text. Keep it natural and descriptive.
- Use the WordPress media library to update alt text so it applies
  everywhere the image is used.

3.2 Images Over 100KB — 96 images (51% of all images on site)
----------------------------------------------------------------------
Issue: Nearly half the images on the site are oversized. This is the
single biggest performance drag on the site. Large images slow page load
significantly on mobile.

Fix:
- WP Rocket is already installed. Go to WP Rocket > Media and enable:
  - LazyLoad for images and iframes
  - "Replace YouTube iframe with preview image" if any YouTube embeds
    are present
- Install and configure Imagify (or use WP Rocket's built-in Imagify
  integration if available on the plan):
  - Bulk optimize all existing images in the media library
  - Set compression to "Aggressive" mode
  - Enable WebP conversion
  - Target: all images should be under 100KB where possible, hero images
    under 200KB
- For any images that remain large after compression (e.g. full-width
  hero photos), resize the actual image dimensions before re-uploading.
  A hero image displayed at 1200px wide does not need to be uploaded at
  4000px wide.
- After bulk optimization, run PageSpeed Insights on the homepage and
  key program pages to confirm improvement.

3.3 Alt Text Over 100 Characters — 3 images
----------------------------------------------------------------------
Fix: Shorten the alt text on these 3 images to be concise. Alt text
should describe the image in one clear phrase, not a paragraph.

SECTION 4 — SECURITY HEADERS (server-level config)
===================================================================

These 5 headers are missing across virtually the entire site (433-434 URLs
affected each). These are all set at the server or .htaccess level — they
do NOT need to be fixed page by page. One change fixes all URLs at once.

Add the following headers to the site's .htaccess file (if Apache) or
the Nginx config (if Nginx). Hosting provider or cPanel access is required.

For Apache (.htaccess), add inside a <IfModule mod_headers.c> block:

  Header always set X-Frame-Options "SAMEORIGIN"
  Header always set X-Content-Type-Options "nosniff"
  Header always set Referrer-Policy "strict-origin-when-cross-origin"
  Header always set Strict-Transport-Security "max-age=31536000; includeSubDomains"
  Header always set Content-Security-Policy "default-src 'self' https: data: 'unsafe-inline' 'unsafe-eval'; frame-ancestors 'self'"

Note on Content-Security-Policy: the value above is a permissive starting
point that should not break the site. A stricter CSP should be configured
and tested separately — a too-strict CSP can break Google Tag Manager,
embedded fonts, or third-party scripts. Set it permissive first, then
tighten.

For Nginx, add to the server {} block in the site config:
  add_header X-Frame-Options "SAMEORIGIN";
  add_header X-Content-Type-Options "nosniff";
  add_header Referrer-Policy "strict-origin-when-cross-origin";
  add_header Strict-Transport-Security "max-age=31536000; includeSubDomains";
  add_header Content-Security-Policy "default-src 'self' https: data: 'unsafe-inline' 'unsafe-eval'; frame-ancestors 'self'";

After adding, verify all headers are present using https://securityheaders.com
or the response headers tab in Chrome DevTools > Network > Headers.

SECTION 5 — LINKS AND NAVIGATION
===================================================================

5.1 Unsafe Cross-Origin Links — 81 pages affected
----------------------------------------------------------------------
Issue: 81 pages have external links that open in a new tab
(target="_blank") without rel="noopener". In older browsers this is a
security vulnerability.

Fix:
- In Elementor, any button or text link that opens in a new tab should
  have rel="noopener noreferrer" added.
- For theme-level navigation links: check the theme's custom link fields.
- This can also be fixed globally with a small snippet in functions.php:

  add_filter('the_content', function($content) {
      return str_replace(
          'target="_blank"',
          'target="_blank" rel="noopener noreferrer"',
          $content
      );
  });

  Note: this only fixes links in post/page content. Links in menus,
  widgets, and Elementor sections need to be updated individually or
  via a more comprehensive filter.

5.2 Internal Nofollow Links — 45 pages affected
----------------------------------------------------------------------
Issue: 45 pages have internal links with rel="nofollow" applied. This
tells Google not to follow those links and not to pass PageRank through
them. On an internal link, this is almost never intentional and wastes
link equity.

Fix:
- Export the Internal Nofollow Outlinks report from Screaming Frog.
- For each nofollow internal link, check if it was intentional. In most
  cases it is not — it is leftover from a plugin, button widget, or
  copied template.
- Remove rel="nofollow" from all internal links where it is not
  deliberately intended. Internal navigation, program page links, and
  CTA buttons should never have nofollow.
- Rank Math Pro may be adding nofollow to certain links automatically —
  check Rank Math > General Settings > Links and review the
  "Add nofollow to external links" and similar settings.

5.3 Internal Redirect — 1 URL
----------------------------------------------------------------------
Issue: 1 internal URL redirects to another URL instead of pointing
directly to the final destination. This creates an unnecessary redirect
hop that slightly slows load time and dilutes PageRank.

Fix:
- Find the redirect URL from the Screaming Frog 3xx Internal report.
- Update the internal link pointing to the redirecting URL to point
  directly to the final destination URL instead.

5.4 External Broken Links (4xx) — 3 external URLs
----------------------------------------------------------------------
Issue: 3 outbound links on the site point to external pages that return
a 4xx error (most likely 404 — page not found on the external site).

Fix:
- Export the External Client Error report from Screaming Frog.
- For each broken external link, either remove the link, update it to
  the correct current URL on the external site, or replace it with a
  relevant working resource.

5.5 Internal Links With No Anchor Text — 81 pages affected
----------------------------------------------------------------------
Issue: 81 pages have internal links where the anchor text is empty. This
happens most commonly with icon links, image links without alt text, or
empty div-based links. Google cannot understand what the linked page is
about when anchor text is missing.

Fix:
- Export the report from Screaming Frog (Bulk Export > Links > Internal
  Outlinks With No Anchor Text).
- For image links: ensure the linked image has descriptive alt text which
  acts as the anchor text.
- For icon-only links (social media icons in the footer, etc.): add an
  aria-label attribute describing the destination, e.g.
  aria-label="VTS on Facebook".
- For any truly empty links: either remove them or add visible or
  screen-reader-only text.

5.6 Non-Descriptive Anchor Text — 1 page
----------------------------------------------------------------------
Issue: 1 page has an internal link with anchor text like "click here"
or "learn more" with no context.

Fix: Update the anchor text to describe the destination page, e.g.
"click here" → "Learn more about our OSHA 30 training program".
